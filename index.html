  • Metaprogramming-based regular expressions in C++20

    Coming soon ...

    Overview

      This project involves writing a custom regular expression parser and matched in C++20. On top of that, the engine must be capable of compiling the regular expression during native code compilation, allowing the user to use them in a constexpr context. This is, of course, achieved through heavy template metaprogramming, inspired by multiple other similar libraries like CTRE or Boost.Xpressive. Basically, this approach makes the regular expression a compile-time known data structure. Instead of creating a std::regex object, which is painfully slow in C++, it is enough to specify a typedef with the C++ using directive, which will alias a particular regular expression. For example, one can write using pattern = meta::regex<"ab*"> to generate the code for matching the regular expression ab*. The generated type contains static methods for performing commong regex operations like matching, searching or iterating through a set of results.

    Matching

      Matching is implemented using the backtracking approach, since this algorithm is capable of matching complex syntactical structures like backreferences and capturing groups. The actual implementation is a bit tricky and uses Continuation-Passing Style techniques to create the necessary control flow, because matching certain elements requires a lot of past and future context (for example, matching (a|ab)+b), which is difficult to directly implement starting from the Abstract Syntax Tree of the regular expression. The main difficulty comes from the fact that it is impossible to dynamically allocate memory in a constexpr function and use it later, at runtime.

    The matching API supports ranges as well as iterator pairs. Any container which satisfies the meta::char_range concept can be matched. That is, it must be a std::ranges::forward_range and its value type must be convertible to char32_t. Matching will return an instance of meta::regex_result, a type which contains information about whether the match succeeded and the captured content (including capturing groups, if present). This type is also explicitly convertible to bool. The library currently supports a lot of the basic PCRE syntax, including capturing groups, backreferences, sets and lookaheads.


  • Discode - chatting application with online code execution

    View on GitHub

    Overview

      Discode (Discord + code, very original, I know) is a web application that has a Discord-like aesthetic and servers as a chatting platform with an integrated online compiler. The application is implemented in a variety of popular Web technologies and allows the user to send a code snippet directly in the chat message, pick a language and execute the code online. Multiple languages are supported, including C++, Python, Rust, Kotlin, Java and Go. The code execution feature is implemented by calling the Compiler Explorer API.

    Architecture

      All the servers can be deployed automatically with docker-compose or manually on localhost by following this guide. There are a total of 4 servers (including the database) and one desktop application written in C++. However, the desktop app is not essential for experiencing the project's features as it is mainly used for user administration.

    The complete list of components is as follows:

    • A MySQL database server, running on port 3435 (in Docker) or 3306 (locally).
    • A Spring Boot backend server on port 8008 written in Kotlin.
    • A Node.js server on port 8010 that uses WebSockets to notify users.
    • An Angular application served by Nginx on port 4200 which is the main web frontend.
    • A QT desktop application written in C++ that is used for administration.
