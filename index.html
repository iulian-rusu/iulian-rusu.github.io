mregex - Metaprogramming-Based Regular Expressions

Overview

  This project involves writing a custom regular expression engine in C++20. What makes this approach special is that the engine is capable of compiling the regular expression into native C++ code, making it much more efficient than standard runtime-interpreted regular expressions. It is also possible to use this kind of regex in a constexpr context, allowing for compile-time matching in cases where the input is a constant expression. This is, of course, achieved through heavy template metaprogramming, inspired by other similar libraries like CTRE or Boost.Xpressive. Basically, this approach makes the regular expression a compile-time known data structure. Instead of creating a std::regex object, which is extremely slow and results in horrible machine code, this library allows mapping a regular expression to a native C++ type. For example, one can write using pattern = meta::regex<"ab*"> to compile the regular expression ab* into a C++ type. The generated type contains static methods for performing common regex operations like matching, searching or iterating through a set of results.

Expression-based API

The library allows defining a regular expression as a composition of C++ expressions. This way, a more complex pattern can be composed from smaller individual patterns. Below is an example of creating the regex (?:\d+[-/.]){2}\d+ using this API: 

// All expression-based API components are in the namespace 'xpr'
auto digits = +xpr::digit;      // Matches one or more digits
auto sep = xpr::regex<"[-/.]">; // Matches '-', '/' or '.'
auto date = xpr::exactly<2>(digits, sep) >> digits;

Matching

  Matching is implemented using the backtracking approach, since this algorithm is capable of matching complex syntactical structures like backreferences and capturing groups. The actual implementation is a bit tricky and uses Continuation-Passing techniques to create the necessary control flow, because matching certain elements requires a lot of future context (for example, matching (a|ab)+), which is difficult to directly implement starting from the Abstract Syntax Tree of the regular expression.

The matching API supports ranges as well as iterator pairs. Any container which satisfies the meta::char_range concept can be matched. That is, it must be a std::ranges::forward_range and its value type must be convertible to char32_t. Matching will return an instance of meta::regex_result, a type which contains information about whether the match succeeded and the captured content (including capturing groups, if present). This type is also explicitly convertible to bool.

The library supports a large portion of the standard PCRE syntax, including features like:

  • Quantifiers - *, +, ?, {3,6}, {5,}
  • Capturing groups - (expr), (?<name>expr)
  • Non-capturing groups - (?:expr)
  • Lookaheads - (?=expr), (?!expr)
  • Lookbehinds- (?<=expr), (?<!expr)
  • Backreferences - \12, \k<name>
  • Alternations - |
  • Special character classes - \w, \d, \S
  • Anchors - ^, $, \b
  • Sets - [abc], [^ABC], [a-z0-9]
  • Flags like icase or multiline


Discode - Chatting Application With Online Code Execution

Overview

  Discode is a web application that has a Discord-like aesthetic (hence its name) and serves both as a chatting platform and as an online code execution engine (hence its name). The application is implemented using a variety of popular Web technologies and allows the user to send a code snippet directly in the chat message, pick a language and run the code online. Multiple languages are supported, including C++, Python, Rust, Kotlin, Java and Go. The code execution feature is implemented by calling the Compiler Explorer API.

Other notable things I've learned by working on this team project include:

  • Designing a distributed Web-service architecture
  • Creating a RESTful API for accesing database resources
  • Performing authentication and authorization using JWT
  • Working with third-party services
  • Automatically deploying servers inside Docker containers
  • Using Git more efficiently to cooperate with my teammates

Architecture

  All the servers can be deployed automatically with docker-compose or manually on localhost by following this guide. There are a total of 4 servers (including the database) and one desktop application written in C++. However, the desktop app is not essential for experiencing the project's features as it is mainly used for user administration.

The complete list of components is as follows:

  • MySQL database server, running on port 3435 (in Docker) or 3306 (locally)
  • Spring Boot backend server on port 8008 written in Kotlin
  • Node.js server on port 8010 that uses WebSockets to notify users
  • Angular application served by Nginx on port 4200 which is the main web frontend
  • QT desktop application written in C++ that is used for administration

Genetic Neural Network Trainer

Overview

  This is an educational project aimed at understanding how neural networks work at a low level, as well as implementing a Genetic Algorithm for training a Multilayer-Perceptron neural network. The project is implemented in C++ and provides a small library abbreviated as gnnt. It contains a custom implementation of a neural network, including activation functions, utilities for dataset manipulation and training metrics. The neural network architecture is defined at compile-time for optimization purposes, using templates to specify the layers with their respective activation functions. The library was designed with genericity and efficiency in mind, supporting custom underlying data types and a variable number of hidden layers, each being specified as a template argument using the gnnt::mlp::layer class template. This has several performance advantages, like better compiler optimizations, avoiding dynamic memory allocations at runtime and the possibility to implement forward propagation using a fold-expression (see the fold() method here.

The main focus of the project was specifically on using a Genetic Algorithm to train the network. That is, each chromosome was represented by the weights and biases of the network, other parameters being taken as constants. As an example of a practical application, the project provides a desktop interface written in QT which uses the library to train a network. The graphical interface provides a 28x28 canvas where the user can draw grayscale images. The ultimate goal is, of course, to recognize the digits drawn using the trained neural network.

Digit Recognition

  The network was trained on the handwritten digits from the MNIST dataset. A simple architecture with two hidden layers of 16 neurons each was able to achieve more than 95% accuracy on the test dataset. Although it is possible, in theory, to further improve this result, the training process was very long due to the size of the chromosomes.

The full source code, as well as the used dataset and serialized models, can be found on GitHub and has the following project structure:

  • data/ - various data used for testing
    • data/mnist/ - dataset for handwritten digit recognition
    • data/models/ - models saved after training
  • gui/ - graphical user interface implemented using QT
  • include/ - main library implementation
    • include/gnnt/mlp/ - compile-time Multilayer Perceptron network
    • include/gnnt/training/- genetic algorithm implementation
  • test/ - test cases for activation functions with the Catch2 framework
